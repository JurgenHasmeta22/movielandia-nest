Section 01

What is Inertia.js?

Inertia.js is a protocol + thin client library that lets you build modern single-page applications using classic server-side routing. Think of it as the glue between a traditional MVC backend (NestJS, Laravel, Rails…) and a modern frontend framework (React, Vue, Svelte).

Without Inertia

  • Backend exposes a REST / GraphQL API
  • Frontend fetches data with useEffect / SWR / TanStack Query
  • You manage loading states, error states, caching
  • Auth tokens passed in headers; CORS config needed
  • Two separate deploy targets

With Inertia

  • Backend routes return page components + props
  • Frontend receives props directly — no fetch calls
  • No loading skeletons: data arrives with the navigation
  • Session-based auth (same as classic MVC)
  • One deploy target, one codebase
The core idea Instead of returning HTML or JSON, every controller action returns an Inertia response: a component name (e.g. "Movies/Index") plus a plain-object payload of props. Inertia serialises this to JSON for XHR visits and wraps it in an HTML shell for full-page loads.

The inertia-nestjs Adapter

The npm package inertia-nestjs is the official NestJS adapter. It provides three things:

Section 03

Request / Response Lifecycle

  1. Browser navigates to /movies — full GET request, no Inertia header yet.
  2. InertiaShareMiddleware runs — reads session, fetches auth.user and flash, merges into shared props for this request.
  3. MovieController.index() executes — queries Prisma, builds { movies, pagination, filters }, returns plain object.
  4. @Inertia("Movies/Index") intercepts the return value — adapter merges page props + shared props into an Inertia page object: { component: "Movies/Index", props: {…}, url: "/movies", version: "…" }.
  5. First load: serialised to JSON inside data-page attribute of <div id="app"> in app.hbs.
  6. React boots — createInertiaApp reads data-page, lazy-imports pages/Movies/Index.tsx, mounts it with the deserialised props.
  7. User clicks a link / router.get() — Inertia intercepts, sends XHR with X-Inertia: true.
  8. Server returns JSON (same shape, no HTML shell), React swaps the component without a full-page reload.
Version checking The adapter compares the X-Inertia-Version header against the server's version (computed from public/build/app.js file size in this project). If they differ, Inertia forces a full-page reload to pick up fresh assets.
Section 04

NestJS Setup

main.ts — bootstrapping the Express host

NestJS sits on top of Express. Three things are wired here for Inertia to work: static assets, the Handlebars view engine, and the session middleware.

src/main.ts
const app = await NestFactory.create<NestExpressApplication>(AppModule);

// Serve Vite-built assets (JS/CSS) from /public
app.useStaticAssets(join(process.cwd(), "public"));

// Handlebars renders the ONE root HTML shell (views/app.hbs)
app.setBaseViewsDir(join(process.cwd(), "views"));
app.setViewEngine("hbs");
hbs.registerHelper("json", (value) => JSON.stringify(value));

// Session — Inertia relies on server-side session for auth + flash
app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false, saveUninitialized: false,
  cookie: { httpOnly: true, secure: isProduction, maxAge: 604800000 },
}));

app.module.ts — wiring InertiaModule

src/app.module.ts
import { InertiaModule } from "inertia-nestjs";

@Module({
  imports: [
    InertiaModule.forRoot({
      rootView: "app",        // → views/app.hbs
      version: assetVersion, // function returning a string for cache-busting
      ssr: {
        enabled: false,
        url: "http://127.0.0.1:13714",
        bundlePath: "bootstrap/ssr/ssr.js",
      },
    }),
    // ... feature modules
  ],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    // Apply the shared-data middleware to every route
    consumer.apply(InertiaShareMiddleware).forRoutes("*");
  }
}

// Version is derived from the built JS bundle's byte length
function assetVersion(): string {
  try { return readFileSync(join(process.cwd(), "public/build/app.js")).length.toString(); }
  catch { return "1.0.0"; }
}

views/app.hbs — the one HTML shell Inertia ever renders

This Handlebars template is rendered only once per full-page load. After that, all navigation is handled by the client-side Inertia router.

views/app.hbs
<!DOCTYPE html>
<html lang="en">
  <head>
    <!-- SSR head tags injected here (title, meta) -->
    {{#each ssrHead}}{{{this}}}{{/each}}

    <!-- Vite-built CSS and JS -->
    <link rel="stylesheet" href="/build/app.css" />
    <script type="module" src="/build/app.js" defer></script>
  </head>
  <body>
    <!-- The "page" variable is the serialised Inertia page object -->
    <!-- React mounts onto this div and reads data-page -->
    {{#if ssrBody}}
      <div id="app" data-page="{{json page}}">{{{ssrBody}}}</div>
    {{else}}
      <div id="app" data-page="{{json page}}"></div>
    {{/if}}
  </body>
</html>
Key insight The {{json page}} helper serialises the Inertia page object (component name + props) as a JSON string into data-page. This is how React knows which component to render and what props to pass — without a separate API call.

The @Inertia() Decorator

The @Inertia(componentName) decorator is all you need to turn any controller method into an Inertia response. Whatever the method returns becomes the props of that page component.

src/app.controller.ts
@Get()
@Inertia("Home")
async home() {
  return await this.appService.getHomeData();
  // → props sent to pages/Home.tsx
}
src/modules/movie/movie.controller.ts
@Get()
@Inertia("Movies/Index")
async index(@Query() query: MovieQueryDto, @Req() req) {
  const data = await this.movieService.findAll(query, userId);
  return {
    movies: data.movies,
    pagination: { total, page, totalPages, perPage },
    filters: query,
  };
}

Full Auth controller — multiple page types

src/auth/auth.controller.ts (excerpts)
@Get("login")
@Inertia("Auth/Login")
loginPage() { return {}; }            // empty props — page needs no server data

@Post("login")
async login(@Body() dto, @Req() req, @Res() res) {
  // POST handlers do NOT use @Inertia — they redirect (PRG pattern)
  const userId = await this.authService.validateCredentials(dto.emailOrUsername, dto.password);
  req.session.userId = userId;
  return res.redirect(303, "/");
}

@Get("reset-password")
@Inertia("Auth/ResetPassword")
resetPasswordPage(@Req() req) {
  return { token: req.query.token ?? "" }; // pass the token as a prop
}
PRG Pattern with Inertia Mutation routes (POST / PUT / DELETE) should not use @Inertia(). Instead, set a flash message on the session and res.redirect(303, …). The redirect target is a GET route decorated with @Inertia(), creating the full POST → Redirect → GET cycle.
Section 06

Shared Data — InertiaShareMiddleware

Some props need to be available on every page — the authenticated user, flash messages, CSRF token, etc. Inertia solves this with the share mechanism. In this project, InertiaShareMiddleware extends HandleInertiaRequests and is applied globally.

src/inertia/inertia-share.middleware.ts
@Injectable()
export class InertiaShareMiddleware extends HandleInertiaRequests {
  constructor(inertia: InertiaService, private readonly prisma: PrismaService) {
    super(inertia);
  }

  async share(req: Request) {
    const base = await super.share(req);      // always call super first
    const userId = req.session?.userId;

    // Fetch auth user from DB on every request
    let authUser = null;
    if (userId) {
      const user = await this.prisma.user.findUnique({
        where: { id: userId },
        select: { id: true, userName: true, email: true, avatar: { select: { photoSrc: true } } },
      });
      if (user) authUser = { id: user.id, userName: user.userName, email: user.email, avatar: user.avatar?.photoSrc ?? null };
    }

    // Read flash once, then delete it (one-time message)
    const flash = req.session?.flash ?? null;
    if (req.session?.flash) delete req.session.flash;

    return { ...base, auth: { user: authUser }, flash };
  }
}
What React components receive Every page automatically receives auth.user and flash alongside its own props. Access them via usePage().props from @inertiajs/react.
One-time flash messages A controller sets req.session.flash = { type: "success", message: "…" }. The middleware reads it and deletes it so it only appears once — exactly like Laravel's flash.

Accessing shared props in React

import { usePage } from "@inertiajs/react";

function Navbar() {
  const { auth, flash } = usePage().props as { auth: { user: User | null }; flash: Flash | null };
  return (
    <nav>
      {auth.user ? <UserMenu user={auth.user} /> : <GuestLinks />}
      {flash && <FlashBanner flash={flash} />}
    </nav>
  );
}
Section 07

Frontend Bootstrap

Client-side entry — resources/js/app.tsx

resources/js/app.tsx
import { createInertiaApp } from "@inertiajs/react";
import { createRoot } from "react-dom/client";

// Vite's import.meta.glob eagerly maps all page files
const pages = import.meta.glob<PageModule>("./pages/**/*.tsx");

createInertiaApp({
  title: (title) => (title ? `${title} - Movielandia24` : "Movielandia24"),

  // Called with the component name from @Inertia("Movies/Index")
  resolve: async (name) => {
    const page = pages[`./pages/${name}.tsx`];
    if (!page) throw new Error(`Page not found: ${name}`);
    const module = await page();
    return module.default;             // React component exported as default
  },

  setup({ el, App, props }) {
    // el = the <div id="app"> from app.hbs
    createRoot(el).render(<App {...props} />);
  },
});
How resolve works Vite pre-maps all pages/**/*.tsx files at build time via import.meta.glob. When Inertia calls resolve("Movies/Index"), you look up "./pages/Movies/Index.tsx" in that map and dynamically import it. The result is code-split automatically — each page is its own JS chunk.

Vite config (bundle output)

vite.config.ts (key parts)
export default defineConfig({
  build: {
    outDir: "public/build",          // served by app.useStaticAssets("public")
    rollupOptions: {
      input: {
        app: "resources/js/app.tsx",  // client entry
        ssr: "resources/js/ssr.tsx",  // SSR entry (optional)
      },
    },
  },
});

React Pages — Consuming Props

A React page is just a default-exported component. Props are passed directly from the controller return value — no hooks, no fetch calls.

resources/js/pages/Movies/Index.tsx
// Props interface matches exactly what MovieController.index() returns
interface MoviesIndexProps {
  movies: Movie[];
  pagination: Pagination;
  filters?: SortFilters;
}

export default function MoviesIndex({ movies, pagination, filters = {} }: MoviesIndexProps) {
  // props arrive directly — no useEffect, no loading spinner needed
  const sortBy = filters.sortBy ?? "title";

  const navigate = (overrides) =>
    router.get("/movies", { sortBy, ...overrides }, { preserveScroll: false });

  return (
    <AppLayout title="Movies">
      <SortControls sortBy={sortBy} onChange={navigate} />
      <MediaGrid items={movies} />
      <PaginationBar {...pagination} />
    </AppLayout>
  );
}

Client-side Navigation

Inertia intercepts clicks on <Link> components and calls to router.*. These become XHR requests instead of full-page reloads.

<Link> component

import { Link } from "@inertiajs/react";

// Renders an <a> tag but intercepts the click
<Link href="/movies">All Movies</Link>
<Link href={`/movies/${id}`}>View</Link>
<Link href="/logout" method="post" as="button">
  Sign Out
</Link>

router programmatic navigation

import { router } from "@inertiajs/react";

// Navigate with query params (sort, page)
router.get("/movies", {
  sortBy: "ratingImdb", ascOrDesc: "desc",
  page: 1, perPage: 12,
}, { preserveScroll: false });

// POST form submission
router.post("/forum/topics", formData);
How Inertia manages browser history Every router.get() or <Link> click pushes or replaces a history entry. The back/forward buttons work natively. The preserveScroll option keeps the scroll position when navigating (useful for filters/sort controls).
Section 10

Error Handling with Inertia

The adapter throws InertiaResponseHandledException after writing its own response. The global exception filter must swallow this — otherwise NestJS would try to write a second response.

src/common/filters/http-exception.filter.ts
@Catch()  // Catch ALL exceptions
export class AllExceptionsFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost): void {

    // ✅ Inertia already sent the response — nothing more to do
    if (exception instanceof InertiaResponseHandledException) {
      return;
    }

    // Handle regular HTTP + custom AppError exceptions normally…
    const { statusCode, message, error } = this.resolveError(exception);
    response.status(statusCode).json({ statusCode, message, error, path, timestamp });
  }
}
Critical gotcha If you forget to handle InertiaResponseHandledException in your filter, every Inertia navigation will throw an unhandled exception error and you will see 500 errors in the console even though the page renders correctly. Always check for it first.
Section 11

Server-Side Rendering (SSR)

Inertia supports SSR via a Node.js micro-server that renders the React component tree to HTML. In this project SSR is configured but disabled (enabled: false). Here is how it works when enabled.

How SSR works

  • The adapter POSTs the page object to the SSR micro-server URL (http://127.0.0.1:13714)
  • The SSR server runs bootstrap/ssr/ssr.js — Vite-built from resources/js/ssr.tsx
  • Returns { head: string[], body: string }
  • Adapter injects ssrHead and ssrBody into the Handlebars template
  • Browser sees real HTML, React then hydrates it
resources/js/ssr.tsx
import createServer from "@inertiajs/react/server";
import ReactDOMServer from "react-dom/server";

createServer((page) =>
  createInertiaApp({
    page,
    render: ReactDOMServer.renderToString,
    resolve: async (name) => {
      const mod = await pages[`./pages/${name}.tsx`]();
      return mod.default;
    },
    setup: ({ App, props }) => <App {...props} />,
  })
);
